I'm super excited to share [transcribe.cpp](https://github.com/handy-computer/transcribe.cpp) today. I've
been working on it for the last handful of months and
it will be powering [Handy](https://handy.computer) going forwards. 
For those wondering, Handy will be updated in the next day (v0.9.0) and 
transcribe.cpp will be powering it.

tldr: transcribe.cpp is a ggml based transcription engine which supports all the latest transcription models. 
every model published under the [handy-comptuer hf org](https://huggingface.co/handy-computer)
has been numerically validated and WER tested to match the reference implementation.

This is a v0.1.0 library which means that there are rough edges. It's good enough
for public release, but there will be issues. 
Please [report them](https://github.com/handy-computer/transcribe.cpp/issues), and let's fix them together.
I hope one day it can be a production grade and highly trustworthy inference engine and
that can only happen with community support.

## Motivation

[Handy](https://handy.computer) is a piece of open source software I wrote
and it has become far more popular than I ever imagined. I had became an open source
maintainer, much to my suprise. This was just a weird piece of software I wanted to share,
but I guess it resonated with people.

And as a result of that, I ended up going further down the transcription rabbit hole than I ever imagined.
And I was already quite familiar with it due to my work on [whisperfile](https://github.com/cjpais/whisperfile).

Let me say this. I think distributing a cross platform application with the current
ASR inference stack is terrible. Pretty much everyone starts with whisper.cpp,
and for good reason. It works. For those developing only for Apple devices,
they have MLX which is excellent. But otherwise you're left on your own, 
maybe picking up a huge variety of libraries, or .onnx files you have no idea
of their provenance. Not to mention many models then became relegated to the CPU
as there was no reasonable way to accelerate them cross platform.

Yet we know we can run most ASR models on Vulkan or Metal via GGML, 
and while the packaging isn't trivial, it is not hard, and everyone 
has done it already.

And this is what led to transcribe.cpp. As Handy's maintainer I needed
a library where I could download a file and run inference on it. It ideally
should run on the GPU for the best performance. Something embeddable in the app.
Not Python. Something that works on Mac, Windows, and Linux. And [ggml](https://github.com/ggml-org/ggml)
seemed like by far the best way forward. It has a strong community, and
a great distribution story.

## So what do you get?

Well, the goal is to be the best distributable transcription engine, and 
one that you can trust. Today that looks like:

* Support for 16 ASR Families (60+ models) with more coming
* Acceleration via Vulkan, Metal, CUDA, and TinyBLAS on CPU
* Support for Streaming Transcription
* Support for Batch Transcription
* Every model has been numerically verified and WER tested
* More or less drop in whisper.cpp replacement
* Maintainer supported bindings in 4 Languages
  - Python
  - Javascript/Typescript
  - Rust
  - ObjC/Swift
* Probably more things I'm forgetting, I'm tired.

## Opinions driving the library

This section is going to be a bit of a ramble.

One of my top goals was to run any ASR model I wanted on Vulkan. In my opinion
this is the floor for any application shipping local inference. So far as I've tested,
this is true for transcribe.cpp. For every model we support, there is 
a corresponding benchmark run from a Ryzen 4750U (CPU + Vulkan) on Fedora.

I also wanted to make sure that inference in transcribe.cpp
is accurate and as close to the reference implementation as possible. 
This largely came from a huge degree of uncertainty of inference accuracy
when using .onnx models I found on Hugging Face. In order to ensure the inference
we do is correct we numerically validate every model versus the reference. 
On top of numerical validation, we run full WER sweeps to make sure that whatever 
the reference is outputting, we output the same thing. That means every model has ran 
through thousands of utterances and is very close or same as the reference. And
the results of this data are published in the transcribe.cpp repo as well as with 
each model on Hugging Face. 

transcribe.cpp is more or less a drop in support for whisper.cpp. The main reason
for this is: Handy used whisper.cpp and I needed to ship an update with 
transcribe.cpp which would replace it. I needed to keep some compatibility
with the very popular .bin files which run in whisper.cpp and shipped with Handy. 
transcribe.cpp can run them. There are probably some flags and features in whisper.cpp
which we do not support. But I think for the vast majority of use cases
our whisper implementation is solid and can replace whisper.cpp while having
about equal performance.

And of course, at the end of the day, a lot of the decisions were driven by Handy.
This library never would have existed without it because I wouldn't have had the 
problem of trying to support a bunch of different ASR models. I would have never 
learned all the use cases that people have for ASR. So, I've done my best to cover 
the ones that I hear about the most. Certainly, there are cases in the 
library that are not currently handled. If there are things that I missed, you 
are free to contribute to the library!

## "Speech-To-Text is a Solved Problem"

The speech-to-text models that we have today are absolutely phenomenal (well for English).
The problem is not with the models themselves, but making them easily accessible to anyone.

transcribe.cpp is aimed squarely at that. We know that transcription can run extremely accurately
on most devices, and there should be no need to send your voice to a cloud service. An RK3566
can run models in transcribe.cpp faster than real time. That is an anemic CPU compared to what 
is in most devices. Faster than real time transcription with SOTA models runs in a handful of watts.
It's not a hope or a dream, it's a fact.

I think as we look forward to the future, more inference will start happening locally
for one reason or the other. This brings the distribution story front and center. 
In order to have more applications running inference locally, we need to make running inference easier. 
Certainly transcribe.cpp does not solve this on the whole, and there is a long way to go, 
but I hope it's a small step forward. I've certainly learned a lot.
